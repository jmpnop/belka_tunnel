# БелкаТуннель: устройство

> 🇬🇧 English version — [README.md](README.md)

**БелкаТуннель** — это нативное приложение macOS в строке меню, которое держит
постоянный SSH-туннель до роутера и отдаёт его локальным программам сразу двумя
способами: как **SOCKS5-прокси** (порт `1080`, для Firefox) и как
**HTTP/HTTPS-прокси** (порт `8080`, для программ, которые умеют только
HTTP-прокси — `curl`, `git`, `npm`, системные поля *Web Proxy* в macOS).
Приложение говорит по SSH само (через библиотеку `russh`), не запуская системный
`ssh`, и грузит ключ `~/.ssh/id_ed25519` напрямую. Скачать `.dmg` можно на
странице [релизов](https://github.com/jmpnop/belka_tunnel/releases) (текущая
версия — `v0.2.0`).

Ниже описано, как та же схема работает «под капотом» на примере ручной команды
`ssh` — приложение делает ровно то же самое, только постоянно и с автоматическим
переподключением.

## Что на компьютере

На компьютере работают два процесса, связанных с этой схемой.

**SSH-клиент.** Запускается командой в терминале:

```
ssh -p 22222 -D 1080 -C -N tunneluser@proxy.example.com
```

Расшифровка флагов:

- `-p 22222` — порт, на котором слушает SSH-сервер на роутере.
- `-D 1080` — открыть на компьютере локальный SOCKS-прокси на порту 1080.
- `-C` — сжимать трафик внутри канала.
- `-N` — не запускать удалённый shell, соединение нужно только ради туннеля.
- `tunneluser@proxy.example.com` — имя пользователя на роутере и имя роутера в DNS. Имя `proxy.example.com` всегда указывает на текущий IP-адрес роутера, даже если провайдер его сменит.

Пока окно открыто, процесс жив и делает две вещи:

- держит постоянное зашифрованное сетевое соединение с роутером по адресу `proxy.example.com` (на момент написания резолвится в `203.0.113.10`), порт `22222`. Аутентификация по ключу `~/.ssh/id_ed25519`, под пользователем `tunneluser`;
- открывает на самом компьютере локальный порт `1080`. Этот порт виден только программам на этом же компьютере. Всё, что попадает на `127.0.0.1:1080`, SSH-клиент берёт, шифрует и отправляет по своему каналу на роутер.

**Firefox.** В настройках сети (*Settings → Network Settings → Manual proxy configuration*) указано:

- SOCKS Host: `127.0.0.1`, Port: `1080`
- SOCKS v5
- Proxy DNS when using SOCKS v5: включено

То есть всё, что Firefox хочет загрузить из сети, он отдаёт на `127.0.0.1:1080`, то есть SSH-клиенту. Остальные программы на компьютере (Telegram, почта, Lightroom) этой настройки не видят и ходят в интернет напрямую.

**HTTP/HTTPS-прокси (только приложение БелкаТуннель).** Кроме SOCKS5, приложение поднимает второй прокси — HTTP на порту `8080`. Он нужен программам, которые не умеют SOCKS, но понимают HTTP-прокси. Тот же туннель, другой «вход»:

```bash
curl -x http://127.0.0.1:8080 https://ifconfig.me      # HTTPS через CONNECT
https_proxy=http://127.0.0.1:8080 curl https://ifconfig.me
curl -x http://127.0.0.1:8080 http://ifconfig.me       # обычный HTTP
curl --socks5-hostname 127.0.0.1:1080 https://ifconfig.me  # тот же результат через SOCKS5
```

Все команды должны вернуть текущий внешний IP роутера. При HTTPS приложение не расшифровывает трафик — браузер/`curl` устанавливает TLS напрямую с сайтом поверх туннеля.

**Доступ с телефона и других устройств в той же сети.** По умолчанию оба прокси слушают на `0.0.0.0`, то есть Mac с БелкаТуннелем становится прокси-шлюзом для всей локальной сети. Телефону или второму компьютеру не нужны ни SSH, ни отдельное приложение — достаточно указать в системных настройках прокси IP-адрес этого Mac в локальной сети.

1. На Mac откройте меню — строки с адресами показывают, что вписывать, например `192.168.1.100:8080 (en0)` для HTTP или `192.168.1.100:1080 (en0)` для SOCKS5. (Если видно только `127.0.0.1 (loopback)` — включите «Listen on all interfaces».)
2. **iPhone/iPad:** *Настройки → Wi-Fi → ⓘ → Настройка прокси → Вручную* → Сервер `192.168.1.100`, Порт `8080`. (В iOS прокси для Wi-Fi только HTTP; HTTPS всё равно работает через CONNECT.)
3. **Другой Mac:** *Системные настройки → Сеть → … → Подробнее → Прокси* → включить *Веб-прокси (HTTP)* и *Защищённый веб-прокси (HTTPS)* → `192.168.1.100:8080`.
4. **Android:** *Wi-Fi → сеть → Дополнительно → Прокси → Вручную* → тот же адрес и порт.

Проверка с телефона: откройте `https://ifconfig.me` — должен показаться внешний IP роутера, а не адрес самого телефона.

> ⚠️ На `0.0.0.0` и без пароля к прокси туннелем может воспользоваться любой, кто дотянется до этого Mac по портам 1080/8080. Для домашней доверенной сети это норма; в чужой/публичной сети выключите «Listen on all interfaces» (только loopback).

## Что на роутере

Роутер — pfSense с внешним IP-адресом, на момент написания `203.0.113.10`. Этот адрес выдан провайдером и может смениться. Чтобы команду на компьютере не приходилось каждый раз переписывать, у роутера есть постоянное имя `proxy.example.com`. Имя зарегистрировано в Cloudflare; сам роутер при изменении своего IP-адреса автоматически сообщает новый адрес в Cloudflare, и имя продолжает указывать на роутер.

На роутере постоянно работает процесс SSH-сервера (sshd), который слушает порт `22222` и ждёт входящие соединения.

Доступ настроен так:

- Пользователь `tunneluser` (uid 2000) с привилегией `user-ssh-tunnel`. pfSense назначает такому пользователю shell `/usr/local/sbin/ssh_tunnel_shell`, при котором проброс портов (`-D`, `-L`, `-R`) работает, а интерактивный вход и запуск команд запрещены.
- Аутентификация только по ключу (`PasswordAuthentication no`). Публичный ключ хранится в `/home/tunneluser/.ssh/authorized_keys`.
- На WAN-интерфейсе (igb0) разрешено правило firewall: вход TCP на порт `22222` со всех источников.

Когда SSH-клиент с компьютера к серверу подключается, между ними устанавливается канал. С этого момента сервер делает следующее:

- получает зашифрованные пакеты от клиента, расшифровывает их и видит исходные запросы Firefox;
- отправляет эти запросы дальше в интернет — со своего IP-адреса `203.0.113.10`, как будто инициатором является роутер;
- получает ответы от сайтов, шифрует их и отправляет обратно по каналу клиенту.

## Что между ними

По сети между компьютером и роутером (`203.0.113.10:22222`) идёт только трафик протокола SSH. SSH сам шифрует всё содержимое: запросы Firefox, ответы сайтов, служебные команды — всё это упаковано внутрь SSH-соединения и по сети передаётся уже в зашифрованном виде. Снаружи (Wi-Fi кафе, провайдер, любое промежуточное оборудование) видно только, что компьютер установил SSH-соединение с адресом `203.0.113.10` на порту `22222`; что именно через это соединение передаётся, посторонним недоступно.

## Что видит сайт

С точки зрения сайта запрос приходит с IP-адреса `203.0.113.10`. О существовании компьютера, с которого всё началось, сайту ничего не известно.

## Когда схема перестаёт работать

- Завершён процесс SSH-клиента на компьютере — порт `1080` закрывается, Firefox получает ошибку «не может подключиться к прокси».
- Роутер `203.0.113.10` выключен или пропал интернет дома — соединение рвётся.
- В настройках Firefox убран прокси `127.0.0.1:1080` — браузер начнёт ходить в интернет напрямую, в обход схемы.

## Автоматическое переподключение

Обычная команда `ssh` не пытается восстанавливать соединение, если оно оборвалось — она просто завершает процесс. Чтобы туннель сам восстанавливался (после смены Wi-Fi, ухода ноутбука в сон, временной пропажи интернета), используется отдельная программа `autossh` или системный механизм `launchd`.

### Вариант 1: `autossh`

`autossh` — обёртка над `ssh`, которая держит SSH-процесс в фоне и перезапускает его, если он умер.

Установка:

```
brew install autossh
```

Запуск:

```
autossh -M 0 -f -N -D 1080 -C \
    -o "ServerAliveInterval 30" \
    -o "ServerAliveCountMax 3" \
    -o "ExitOnForwardFailure yes" \
    -p 22222 tunneluser@proxy.example.com
```

Что добавилось к исходным флагам:

- `-f` — уйти в фон сразу после установления соединения; терминал закрывать можно.
- `ServerAliveInterval 30` — каждые 30 секунд посылать через канал служебный «пинг»; если ответа нет три раза подряд (`ServerAliveCountMax 3`), SSH считает соединение мёртвым и завершается. После этого `autossh` запускает его заново.
- `ExitOnForwardFailure yes` — если при подключении не удалось открыть локальный порт `1080`, не запускать туннель в «пустом» состоянии, а завершиться. autossh попробует ещё раз позже.

После запуска можно проверить, что процесс жив:

```
pgrep -fl autossh
```

Остановить:

```
pkill -f autossh
```

### Вариант 2: `launchd` (без сторонних программ)

`launchd` — встроенная в macOS система запуска фоновых задач. Можно описать «всегда держи этот процесс запущенным; если он умер — перезапусти; запускай при логине».

Файл `~/Library/LaunchAgents/com.client.ssh-proxy.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key><string>com.client.ssh-proxy</string>
    <key>ProgramArguments</key>
    <array>
        <string>/usr/bin/ssh</string>
        <string>-N</string>
        <string>-D</string><string>1080</string>
        <string>-C</string>
        <string>-p</string><string>22222</string>
        <string>-o</string><string>ServerAliveInterval=30</string>
        <string>-o</string><string>ServerAliveCountMax=3</string>
        <string>-o</string><string>ExitOnForwardFailure=yes</string>
        <string>tunneluser@proxy.example.com</string>
    </array>
    <key>KeepAlive</key><true/>
    <key>RunAtLoad</key><true/>
    <key>StandardOutPath</key><string>/tmp/ssh-proxy.log</string>
    <key>StandardErrorPath</key><string>/tmp/ssh-proxy.err</string>
</dict>
</plist>
```

Загрузка (после первого создания файла или после правок):

```
launchctl unload ~/Library/LaunchAgents/com.client.ssh-proxy.plist 2>/dev/null
launchctl load   ~/Library/LaunchAgents/com.client.ssh-proxy.plist
```

Что делает `launchd`:

- `RunAtLoad` — запустить процесс сразу после загрузки конфигурации (и автоматически после логина).
- `KeepAlive` — если процесс завершится (по любой причине: убит, упал, оборвался ssh), запустить заново.
- Логи пишутся в `/tmp/ssh-proxy.log` и `/tmp/ssh-proxy.err`, можно смотреть `tail -f /tmp/ssh-proxy.err`.

Остановка и выгрузка:

```
launchctl unload ~/Library/LaunchAgents/com.client.ssh-proxy.plist
```

### Что важно понимать

Автоматическое переподключение не делает туннель «вечным» — оно только избавляет от необходимости каждый раз вручную запускать команду заново. Если интернета нет, или роутер дома выключен, или ключ перестал подходить — переподключение будет циклически не удаваться (что видно в логах `/tmp/ssh-proxy.err`). Как только условия восстановятся, следующая попытка пройдёт успешно.
